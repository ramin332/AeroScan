# Reviewing an AeroScan custom PSDK WaypointV3 flight after it lands

**Investigation date:** 2026-06-12
**Scope:** read-only. How to access/review a custom inspection flight (the M4E flown via `DjiWaypointV3_UploadKmzFile` + `Action(START)` from the Manifold PSDK app).
**Subject flight:** `flight0065` scan → `20260612T122123Z_441` augmented mission → flown and recorded in **`flight0066`** PSDK log (581 WPs, Mijande/Amsterdam, 52.4198 N 4.9513 E).

---

## TL;DR / recommendation

The pilot's complaint ("our flight stays on the drone, no way to review it") is **half right**. We do **not** lose the flight — but the review path is different from a DJI-native mission, and nobody has wired it up yet.

Two things survive every custom flight and together let us reconstruct map + trajectory + gimbal:

1. **The flown KMZ** (`*.augmented.lean.kmz`) persists on the Manifold under `/open_app/dev/data/received/`. Its `waylines.wpml` is the full per-waypoint **intended** trajectory: lon/lat, height, aircraft heading, gimbal pitch — 581 waypoints for our flight. The aircraft flew this autonomously.
2. **The Manifold PSDK log** (`/blackbox/<flight>/psdk/PSDK-*.log`, plaintext) records the **actual execution**: every `mission state` and `action state` transition with timestamps and waypoint index — proof the aircraft progressed through (and where it paused) the route.

**Recommendation (do both, in order):**

- **Now / zero-build:** also pull the **DJI flight record `.txt` off the RC Plus 2** (Pilot 2 records it during our flight because Pilot 2 stays live as the monitor). View it in any free DJI log viewer (Phantom Help / Airdata / Flight Reader). This is DJI's *ground-truth* aircraft telemetry (GPS/altitude/attitude/gimbal at ~10 ms) and is the fastest answer to "did it actually fly where we told it." **This needs one on-RC verification we have not done** — flagged below as a GO/NO-GO.
- **Small build (~1–1.5 days):** an **AeroScan flight-review view** that ingests the flown `waylines.wpml` + the PSDK-log state timeline and renders the planned route + a time-scrubbed drone on the existing Three.js/Leaflet viewers. Reuses `DroneAnimation.tsx` / `Viewer3D.tsx` / `MapView.tsx` almost wholesale. This is the durable, in-product answer and does not depend on DJI's encrypted record.

## 1. Does a PSDK WaypointV3 mission produce a DJI flight record?

**Almost certainly yes — on the RC, not on the Manifold — but with one unverified step.**

- A DJI flight record is generated by the **RC app (Pilot 2)**, which logs the aircraft's downlinked telemetry stream (position/speed/altitude/attitude/gimbal at ~10 ms) into an encrypted `.txt`. It is created for *any* flight where Pilot 2 is the live monitor — the record is keyed to the telemetry link, **not** to who issued the flight command.
- Our architecture keeps **Pilot 2 live as the monitor** during the custom flight (it shows FPV + telemetry + our widget; only the *mission player* is bypassed — see `aeroscan-psdk/docs/psdk/api-notes.md` and `architecture/kmz-flow.md`). So the telemetry link is up the whole flight and Pilot 2 has every reason to be writing a normal flight record.
- **Where on the RC Plus 2:** `Internal shared storage/DJI/com.dji.industry.pilot/FlightRecord/` (and `MCDatFlightRecords/` for the FC `.DAT`). Copy off via USB-MTP, exactly like the existing manual Smart3D-KMZ export step.

**Unverified (GO/NO-GO for option (a)):** we have **not** confirmed on-device that a `.txt` actually appears in that folder after a PSDK WaypointV3 flight, nor that it contains the full trajectory (vs. a stub). It is plausible DJI tags PSDK-commanded flights differently or thins the record. **First action on the next field day: fly/replay a short custom mission, then check `FlightRecord/` on the RC for a new `.txt` and decode it.** Treat as a 5-minute check, not a guarantee.

**On the Manifold:** there is **no** DJI flight record. The `/blackbox/<flight>/` tree is encrypted DJI internals plus our own PSDK log — see §2.

## 2. Where the per-flight telemetry lives, and what's in it

Three independent stores. Only two are ours to read today.

### (A) Manifold `/blackbox/<flight>/` — mostly encrypted, one gem

Per-flight dir layout (verified against the downloaded `flight0064–0067` and live `192.168.1.55`):

| Subdir / file | Format | Encrypted? | Recoverable content |
|---|---|---|---|
| `psdk/PSDK-NNNN-01.log` | **plaintext** text log | **No** | ⭐ **Our mission execution timeline** — see below |
| `dji_perception/1/mesh_binary_*.ply` | binary PLY | No | Building point cloud (only if a Smart3D scan ran that session). Not flight telemetry. |
| `dji_perception/1/*.enc` (`expl_plan.bin.enc`, `vp.log.enc`, `curve.enc`, `raw_data.enc`…) | DJI proprietary | **Yes** | Opaque. Flight plan, VIO/SLAM, raw sensor — all sealed. |
| `dji_mcu/MCU-*.log.enc` | DJI proprietary | **Yes** | MCU/FC log — opaque. |
| `diag/diag-*.log.enc` | DJI proprietary | **Yes** | Diagnostics — opaque (22 MB on flight0066). |
| `camera/` | — | — | **Empty.** Photos do NOT land here (they go to the camera SD card / `/data/image`). |
| `system/` | — | — | **Empty.** |
| `*_storage.json`, `Smart3DExplore_dynamic_ch_v2.json` | JSON | No | Storage/IPC config only — no trajectory. |

**The PSDK log is the recoverable per-flight telemetry on the Manifold.** For `flight0066` it captured the entire custom flight:

```
14:24:40  DjiWaypointV3_UploadKmzFile OK — 128770 bytes from .../20260612T122123Z_441.augmented.lean.kmz
14:24:40  Fly — DjiWaypointV3_Action(START)
14:24:42  Waypoint v3 mission state: 16  (preparing) — waypoint 1
   …      mission state / action state transitions, timestamped, with waypoint index + action group/id
14:42:11  Waypoint v3 mission state: 64  (paused) — waypoint 288
14:42:15  Waypoint v3 mission state: 0   (idle)
```

What the PSDK log **does** give: timestamped mission lifecycle (idle/preparing/flying/paused), **per-waypoint progress index** (1→288 observed), per-action-group execution (camera/gimbal trigger events), upload/START success, error codes on refusal.

What it **does NOT** give: **no GPS, no altitude, no aircraft attitude, no gimbal angle.** This is a hard limit of the API, not a logging gap — the WaypointV3 callbacks only carry indices:

```c
// dji_waypoint_v3.h (psdk_lib)
typedef struct { E_DjiWaypointV3MissionState state; uint16_t currentWaypointIndex; uint32_t wayLineId; } T_DjiWaypointV3MissionState;
typedef struct { E_DjiWaypointV3ActionState state; uint32_t wayLineId; uint16_t currentWaypointIndex,
                 actionGroupId, actionId; } T_DjiWaypointV3ActionState;
```

Our app (`kmz_runner.c`) registers both callbacks and logs them verbatim. It does **not** subscribe to `DjiFcSubscription` (no TOPIC_GPS / position / quaternion / gimbal) — so the Manifold side never sees the aircraft's real position. (That's a cheap future enhancement — see §3.)

### (B) Flown KMZ on the Manifold — the trajectory source (intended)

`/open_app/dev/data/received/20260612T122123Z_441.augmented.lean.kmz` (the exact 128 770-byte file the log shows uploaded) is still on disk. Contents:

- `wpmz/waylines.wpml` — **581 waypoints**, each: `<coordinates>lon,lat</coordinates>`, `<wpml:executeHeight>`, `<wpml:aircraftHeading>` (yaw), `<wpml:gimbalPitchRotateAngle>`. `autoFlightSpeed=3.0`.
- `wpmz/res/.../sfm_geo_desc.json` — `LOCAL_ENU_CS` georeference (`ref_GPS` lat/lon/alt) tying the local frame to WGS84.
- A `*.summary.json` sidecar with gimbal stats (pitch min/median/max, anomaly indices), ICP alignment metrics, WP/facade counts.

This is the **intended** route the aircraft autonomously executed. It is *not* a measured record — but the M4E flies WaypointV3 to cm-class accuracy, so for "review what we flew" it's faithful. Pair it with the PSDK-log index timeline to know **when** the aircraft was at each WP and where it paused.

### (C) RC Plus 2 DJI flight record `.txt` — ground truth (off the Manifold)

The only store with **measured** aircraft telemetry (GPS/altitude/attitude/gimbal at ~10 ms). Encrypted DJI `.txt`, decodable by mature free tools (Phantom Help Log Viewer, Airdata, Flight Reader, dji-log-parser). Lives on the **RC**, reached by USB-MTP — same workflow as the existing Smart3D-KMZ export. **Existence-for-PSDK-flights is the §1 GO/NO-GO check.**

## 3. Practical path to VIEW/REPLAY — options evaluated

### (a) DJI's native record on the RC — **lowest effort, ground truth, but unverified + out-of-product**
- **Effort:** ~0 build. Copy `.txt` off RC, drag into a free viewer → map + trajectory + gimbal + battery.
- **Pros:** real measured telemetry; richest data; zero engineering.
- **Cons:** (1) **unconfirmed that a usable record exists for PSDK WaypointV3 flights** — must verify on-RC first; (2) lives outside AeroScan — pilot juggles another tool; (3) DJI-encrypted, third-party decoders can break across firmware.
- **Verdict:** do the verification immediately; if it exists, use it as the authoritative cross-check. Not a product feature, a forensic tool.

### (b) Third-party DJI log decoders on the Manifold `.enc` files — **dead end**
- The Manifold's `dji_perception/*.enc`, `dji_mcu/*.enc`, `diag/*.enc` are DJI-proprietary encrypted; no public decoder. Prior on-device investigation (`data-locations.md`, 2026-05-03) already closed the search for plaintext flight data on the Manifold. The only DJI logs a decoder *can* touch are the RC-side `.txt`/`.DAT` — which is option (a), not this.
- **Verdict:** skip.

### (c) Build a small AeroScan flight-review view — **recommended product answer**
Render the **flown `waylines.wpml`** (planned route + gimbal) and overlay the **PSDK-log state timeline** (actual progress/pauses) on our existing viewers.

- **Data we already have, no new on-drone work:** the lean KMZ (`/open_app/dev/data/received/`) + the PSDK log (`/blackbox/<flight>/psdk/`). Both are already pulled to the laptop in this archive; both are scp-able live.
- **Reuse:** the frontend already animates a drone over a mission path — `DroneAnimation.tsx`, `Viewer3D.tsx` (Flight mode), `MapView.tsx` (Leaflet + path). A WPML parser is trivial (we already emit WPML in `kmz_builder.py`; reverse is ~an afternoon) and the `sfm_geo_desc.json` `ref_GPS` gives the ENU↔WGS84 tie the viewers expect.
- **What it shows:** map + 3D trajectory, per-WP gimbal pitch/yaw arrows (we already render `OriginalGimbalArrows`), and a scrubber driven by PSDK-log timestamps so you can replay "WP 1 … paused at 288." Flag the 13 `pitch_down` anomaly WPs from `summary.json`.
- **Effort:** **~1–1.5 days.** WPML→mission parser (0.5d) + a Flight-Review tab wiring the existing viewers + a timeline scrubber from the PSDK-log states (0.5–1d). No backend pipeline changes.
- **Optional follow-on (~0.5d) — close the "intended vs actual" gap on the Manifold side:** add a `DjiFcSubscription` to `kmz_runner.c` for TOPIC position + gimbal and append it to the PSDK log (or a sidecar CSV) during flight. Then the review uses *measured* aircraft telemetry without ever touching the RC's encrypted record. This is the cleanest long-term fix and removes the dependency on option (a) entirely.

**Recommended combination:**
1. **Immediately (5 min, field):** verify the RC `FlightRecord/.txt` exists for a PSDK flight and decodes (option a). Settles GO/NO-GO and gives a ground-truth reference.
2. **~1–1.5 days:** build the AeroScan Flight-Review tab from the flown KMZ + PSDK log (option c). This is what the pilot actually wants — review *inside* AeroScan, no extra tools.
3. **+~0.5 day, when convenient:** add FC-subscription logging on the Manifold so the review shows measured (not just intended) telemetry and is fully self-contained.

## Hard facts established (for whoever builds this)

- The flown mission KMZ persists at `/open_app/dev/data/received/<ts>_<id>.augmented.lean.kmz` and is **not** churned by the `/blackbox` ring buffer (different filesystem, our directory). Back it up — it's the only copy of the intended trajectory once `/blackbox` cycles.
- `/blackbox` IS a ~30-slot ring buffer (matches existing memory `manifold-mesh-volatile`). The **PSDK log inside it is volatile** — archive `psdk/PSDK-*.log` alongside the KMZ for any flight you want to review later. (This archive already did for `flight0064–0067`.)
- WaypointV3 state callbacks expose **index only** — position/attitude/gimbal must come from the KMZ (intended) or an FC subscription / RC record (measured). There is no position in the Manifold logs today.
- `camera/` and `system/` under `/blackbox/<flight>/` are empty; geotagged photos are on the camera SD card, not here. JPEG XMP (`drone-dji:GimbalPitchDegree` etc.) remains the per-photo ground truth if image-level review is ever needed.
